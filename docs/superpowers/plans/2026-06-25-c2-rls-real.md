# C2 — Plan de implementación: Migración a RLS real

> **Para Hermes:** ejecutar tarea por tarea. Cambios de código → Claude Code CLI. DDL/DB → vía psql con backup. Validar en STAGING antes de PROD.

**Spec:** `docs/superpowers/specs/2026-06-25-c2-rls-real-design.md`
**Goal:** Que el backend se conecte como `pqrs_backend` (RLS activo) y el aislamiento multi-tenant lo garantice Postgres, sin romper ningún flujo legítimo.
**Arquitectura:** El andamiaje RLS ya existe (código inyecta contexto + rol + 10/20 policies). Este plan: cierra 4 huecos de policy + 2 casos especiales, valida en staging poblado, y migra prod con rollback de 1 env var.
**Stack:** PostgreSQL RLS, FastAPI/asyncpg, Docker Compose (prod `pqrs_v2_*`, staging `pqrs_staging_*`).

## Hechos verificados en prod (2026-06-25) — NO re-verificar
- Rol `pqrs_backend` existe: `rolsuper=f, rolbypassrls=f`, con GRANTs CRUD sobre las 20 tablas **y USAGE sobre las 2 secuencias** (INSERTs OK).
- 10/20 tablas con RLS+policy compatible (`current_setting('app.current_tenant_id')` + bypass `is_superuser`).
- 4 tablas multi-tenant SIN policy: `audit_log_respuestas` (sin `cliente_id`, tiene `caso_id` → policy por JOIN), `borrador_feedback` / `kb_ingestion_log` / `pqrs_clasificacion_feedback` (tienen `cliente_id` → policy directa).
- `plantillas_respuesta`: 62 filas, todas con `cliente_id` → por-tenant → lleva policy directa.
- `clientes_tenant`: PK `id` ES el tenant → policy especial `id = current_tenant OR is_superuser`.
- Globales sin RLS (correcto, NO tocar): `festivos_colombia`, `aequitas_migrations(_lock)`.
- `pqrs_comentarios_bak_20260625`: backup temporal → BORRAR en Tarea 1.
- **STAGING está desalineado:** DB `pqrs_staging`, NO tiene rol `pqrs_backend`, solo 6 policies, y está casi vacío (1 tenant, 0 casos, 1 usuario) → requiere preparación + seed antes de testear.
- Backend prod tiene `pytest` + `/app/tests/conftest.py` + `pytest.ini` (la suite corre con asyncio_mode correcto).

## FASE 1 — Policies en STAGING (validación segura)

### Tarea 3: Alinear el rol pqrs_backend en staging
**Objetivo:** que staging tenga el mismo rol no-superuser que prod.
**Pasos:**
1. `CREATE ROLE pqrs_backend LOGIN PASSWORD '<staging-pass>' NOSUPERUSER NOBYPASSRLS;` en `pqrs_staging`.
2. `GRANT SELECT,INSERT,UPDATE,DELETE ON ALL TABLES IN SCHEMA public TO pqrs_backend;`
3. `GRANT USAGE,SELECT ON ALL SEQUENCES IN SCHEMA public TO pqrs_backend;`
4. `ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT,INSERT,UPDATE,DELETE ON TABLES TO pqrs_backend;`
**Verificación:** `SELECT rolname,rolbypassrls FROM pg_roles WHERE rolname='pqrs_backend'` → `f`.

### Tarea 4: Sembrar 2 tenants con datos en staging
**Objetivo:** poder probar aislamiento real (hoy staging tiene 1 tenant, 0 casos).
**Pasos:** insertar 2 tenants (A y B), 1 usuario admin por tenant, y 2-3 casos por tenant con adjuntos/comentarios. (Datos sintéticos, sin PII real.)
**Verificación:** `SELECT cliente_id, count(*) FROM pqrs_casos GROUP BY 1` → 2 tenants con casos.

### Tarea 5: Aplicar las policies nuevas en staging (las 4 + casos especiales)
**Objetivo:** dejar staging con el set COMPLETO de policies que irá a prod.
**Pasos (un archivo SQL `c2_policies.sql`, idempotente con `DROP POLICY IF EXISTS`):**
- `borrador_feedback`, `kb_ingestion_log`, `pqrs_clasificacion_feedback`, `plantillas_respuesta` → ENABLE RLS + policy directa por `cliente_id`.
- `audit_log_respuestas` → ENABLE RLS + policy por JOIN a `pqrs_casos`.
- `clientes_tenant` → ENABLE RLS + policy especial `id = current_tenant OR is_superuser`.
**Verificación:** `SELECT count(*) FROM pg_policies WHERE schemaname='public'` aumentó en 6; cada policy usa `app.current_tenant_id`.

### Tarea 6: Cambiar backend de staging a pqrs_backend + restart
**Objetivo:** activar RLS en el backend de staging.
**Pasos:** editar `DATABASE_URL` del servicio `pqrs_staging_backend` (compose/env) → `pqrs_backend`; restart.
**Verificación:** backend staging HTTP 200; log sin errores de permisos.

### Tarea 7: Matriz de testing por rol en staging (EL NÚCLEO)
**Objetivo:** probar que ningún flujo legítimo se rompe y que el cross-tenant queda bloqueado.
**Sub-tests (cada uno con su criterio):**
- **Login + /auth/me** para admin de tenant A y de tenant B → 200, devuelve SU tenant. (Valida `clientes_tenant`.)
- **Bandeja / listado de casos** admin A → solo casos de A; admin B → solo de B.
- **Detalle de caso** A pide caso de A → OK; A pide caso de B → 404/0 filas.
- **Descarga de adjunto** cross-tenant → bloqueado (ya cubierto por C1, revalidar bajo RLS).
- **Envío de lote** (mock del proveedor) → escribe en `audit_log_respuestas` y se lee correctamente (valida policy por JOIN).
- **Stats/dashboard** A → números solo de A.
- **Plantillas** A → solo plantillas de A.
- **super_admin** → ve todo (los 2 tenants).
- **Creación de caso/comentario** → INSERT OK (valida secuencias + policy INSERT).
- **Suite existente**: `pytest /app/tests/ -q` con `pqrs_backend` → sin regresiones nuevas.
**Verificación:** TODOS los sub-tests pasan. Cualquier "0 filas inesperado" en un flujo legítimo = bug de policy → corregir antes de avanzar. NO se pasa a Fase 2 sin verde total.

## FASE 2 — Deploy a PROD (quirúrgico, tras staging verde)

### Tarea 8: Aplicar policies en prod
**Pasos:** backup esquema (Tarea 1 ya hecho), aplicar el mismo `c2_policies.sql` validado en staging sobre `pqrs_v2`. Confirmar GRANTs/secuencias de `pqrs_backend` (ya verificados, re-chequear USAGE).
**Verificación:** `pg_policies` prod = 16 policies (10 previas + 6 nuevas); el backend AÚN como `pqrs_admin` (todavía no cambia) → todo sigue funcionando igual (las policies nuevas no afectan a un superuser).

### Tarea 9: Cambiar backend de prod a pqrs_backend
**Pasos:** editar `DATABASE_URL` de `pqrs_v2_backend` → `pqrs_backend` (env/compose, NO hardcode). `docker restart pqrs_v2_backend`.
**Verificación:** HTTP 200, startup complete, log sin errores de permiso. Smoke test inmediato: login + bandeja de un usuario real de cada tenant + un detalle de caso. Si algo da vacío → **ROLLBACK** (Tarea 10).

### Tarea 10: Plan de rollback (tener listo, ejecutar solo si falla T9)
**Pasos:** revertir `DATABASE_URL` a `pqrs_admin` + `docker restart pqrs_v2_backend`. Las policies nuevas quedan (no molestan a superuser). Si una policy específica rompiera algo incluso con superuser (no debería), `DROP POLICY <nombre>`.
**Verificación:** sistema vuelve al estado pre-C2 en <1 min.

### Tarea 11: Actualizar el default de código + commit
**Pasos:** vía Claude Code, actualizar `config.py:5` default a `pqrs_backend` (coherencia; el valor real viene de env igual). Actualizar Brain (bitácora + cerrar C2 en informe maestro). Mergear PRs.
**Verificación:** `git diff` = solo esa línea; informe maestro marca C2 resuelto.

## Orden de ejecución y gates

## Riesgos residuales
- Si un endpoint hace queries que el código NO filtra y que dependían del bypass → 0 filas. Mitigado por Tarea 7 (matriz por rol). El testing en staging poblado es la red principal.
- `set_config(is_local=false)` reusa contexto en el pool → mitigado por la limpieza en `finally` de `db.py`. Mejora a `SET LOCAL` queda fuera de alcance (anotada en spec).
